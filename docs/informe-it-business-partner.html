Te presento formalmente el proyecto Album CDT Mundial 2026, una aplicación web interna desarrollada para la campaña de incentivos de ventas de la compañía durante el Mundial de Fútbol 2026. El documento cubre arquitectura, seguridad, costos y consideraciones operativas para su evaluación y soporte.

1. Descripción del Proyecto

El sistema es un álbum de figuritas digital gamificado. Durante 6 semanas (alineadas con las fases del Mundial), los empleados que cumplan su meta de ventas desbloquean una figurita con su foto en el álbum de su tienda. El objetivo es aumentar el cumplimiento de metas y generar reconocimiento entre colaboradores.

Actores del sistema

  • Administrador central: configura tiendas, sube resultados de ventas, gestiona empleados y avanza semanas.
  • Jefe de tienda: accede con código + contraseña para ver el álbum de su tienda.
  • Automatización (Apps Script): sincronización semanal de resultados de ventas desde Google Sheets.
Alcance temporal: Campaña activa del 9 de junio al 19 de julio de 2026 (6 semanas). Posterior al Mundial, la infraestructura puede pausarse o eliminarse.

5. Seguridad y Autenticación

Flujo de autenticación

  1. La tienda envía código + contraseña al endpoint POST /api/auth/login.
  2. El backend verifica el hash bcrypt en Cloud SQL.
  3. Si es correcto, el servidor genera un Firebase Custom Token firmado con la Service Account.
  4. El frontend lo intercambia por un Firebase ID Token (válido 1 hora, renovación automática).
  5. Cada request subsecuente incluye el ID Token en el header Authorization: Bearer <token>.
  6. El backend llama admin.auth().verifyIdToken() para validar la firma y los claims.

Apps Script (Google Sheets)

Usa una API Key independiente en el header x-api-key para las rutas de sincronización. No usa Firebase Auth — es una integración server-to-server interna.

Prácticas de seguridad aplicadas

  • Contraseñas almacenadas con bcrypt (10 rondas salt).
  • CORS configurado exclusivamente para el dominio de Firebase Hosting.
  • Variables de entorno sensibles gestionadas en Cloud Run secrets (no en código).
  • Service Account con permisos mínimos: Cloud SQL Client, Storage Object Admin, Firebase Admin.
Recomendación pendiente: Migrar la clave privada de la Service Account a Google Secret Manager para la siguiente iteración. Actualmente se gestiona como variable de entorno en Cloud Run.

10. Consideraciones para IT

Respaldo de datos

Se recomienda activar backups automáticos diarios en Cloud SQL desde la consola de GCP. La ventana de retención sugerida es 7 días durante la campaña.

Monitoreo

Cloud Run expone métricas de latencia, errores y uso de CPU en Cloud Monitoring. No se ha configurado alerting formal — se recomienda agregar una alerta de error rate > 5% a un correo de IT.

Rotación de credenciales

La clave privada de la Service Account debe rotarse periódicamente desde IAM & Admin → Service Accounts en GCP Console.

Acceso de emergencia

  • URL de la aplicación: https://album-metat-mundial-59cad.web.app
  • Panel admin: /admin.html
  • Consola GCP: proyecto album-metat-mundial-59cad
  • Las credenciales de administrador residen en la tabla admins de Cloud SQL.

Sin dependencias de terceros críticos

Todo el stack reside en GCP / Google Cloud. No hay dependencias en servicios externos pagos o de terceros fuera del ecosistema Google.
