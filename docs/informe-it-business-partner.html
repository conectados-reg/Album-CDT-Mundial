Presentamos formalmente el proyecto Album CDT Mundial 2026, una aplicación web interna desarrollada para la campaña de incentivos de ventas de la compañía durante el Mundial de Fútbol 2026. El documento cubre arquitectura completa, seguridad, integraciones, costos y consideraciones operativas.

1. Descripción del Proyecto

El sistema es un álbum de figuritas digital gamificado. Durante 6 semanas (alineadas con las fases del Mundial), los empleados que cumplan su meta de ventas desbloquean una figurita con su foto en el álbum de su tienda. Al desbloquearse, el sistema envía automáticamente un correo de notificación al jefe de tienda para que suba la fotografía del empleado.

Actores del sistema

  • Administrador central: configura tiendas, carga resultados de ventas, gestiona empleados y avanza semanas.
  • Jefe de tienda: accede con código + contraseña para ver el álbum de su tienda y subir fotos de empleados.
  • Apps Script (Google Sheets): sincronización semanal masiva de resultados de ventas al sistema.
  • Brevo: servicio externo de envío de correos transaccionales de notificación.
Alcance temporal: Campaña activa del 9 de junio al 19 de julio de 2026 (6 semanas). Al finalizar el Mundial, la infraestructura puede pausarse o eliminarse.

5. Integraciones Externas

Google Sheets + Apps Script

El proceso de carga de resultados opera de dos maneras:

  1. Apps Script: un script en el archivo de Google Sheets llama directamente los endpoints /api/sync/registrar-tienda y /api/sync/resultados mediante HTTP, autenticado con la SYNC_API_KEY.
  2. Sheets API directa: el backend también puede leer el Google Sheet directamente usando la librería googleapis de Node.js, autenticado con la Service Account de GCP, para lectura de la pestaña de ventas (Ventas!A2:E).

Brevo (Email transaccional)

Cuando el administrador confirma el desbloqueo de figuritas, el sistema genera un correo HTML personalizado con el nombre de la tienda, la semana y las tarjetas de los empleados que cumplieron meta. El correo se envía via Brevo REST API (api.brevo.com/v3/smtp/email) al email registrado de la tienda. El remitente aparece como "Álbum Estrellas · SLA Corp.".

6. Seguridad y Autenticación

Flujo de autenticación

  1. La tienda envía código + contraseña al endpoint POST /api/auth/login.
  2. El backend verifica el hash bcrypt (10 rondas) en Cloud SQL.
  3. Si es correcto, el servidor genera un Firebase Custom Token firmado con la Service Account.
  4. El frontend lo intercambia por un Firebase ID Token (válido 1 hora, renovación automática).
  5. Cada request incluye el ID Token en el header Authorization: Bearer <token>.
  6. El backend llama admin.auth().verifyIdToken() para validar firma y claims de rol.

Apps Script

Usa una API Key independiente en el header x-api-key. No usa Firebase Auth.

Prácticas de seguridad aplicadas

  • Contraseñas almacenadas con bcrypt (10 rondas salt).
  • CORS configurado exclusivamente para el dominio de Firebase Hosting.
  • Variables de entorno sensibles gestionadas como secrets en Cloud Run (no en código fuente).
  • Service Account con permisos mínimos: Cloud SQL Client, Storage Object Admin, Firebase Admin.
  • Fotos de empleados almacenadas en bucket privado con URL pública por objeto (no acceso de lista).

11. Consideraciones para IT

Respaldo de datos

Se recomienda activar backups automáticos diarios en Cloud SQL desde la consola de GCP, con retención de 7 días durante la campaña.

Acceso de emergencia

  • URL de la aplicación: https://album-metat-mundial-59cad.web.app
  • Panel administrador: /admin.html
  • Consola GCP: proyecto album-metat-mundial-59cad
  • Credenciales de admin en la tabla admins de Cloud SQL.

Sin dependencias de terceros críticos en GCP

Todo el stack principal (frontend, backend, base de datos, almacenamiento, autenticación) reside en Google Cloud / Firebase. La única dependencia externa es Brevo para el envío de emails, cuya caída no afecta la operación del álbum ni el registro de ventas.
